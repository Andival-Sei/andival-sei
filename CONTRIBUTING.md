# Руководство по работе с Git Flow

Данный документ описывает процесс разработки в проекте Andival SEI.

## 🌿 Структура веток

### Основные ветки

- **`main`** - продакшн-ветка, содержит только стабильный код, готовый к деплою
- **`develop`** - ветка разработки, содержит последнюю версию разрабатываемых функций

### Временные ветки

Все ветки фич создаются от `develop` и удаляются после мерджа через PR.

## 📋 Workflow

### 1. Создание ветки для новой функции

Всегда создавайте ветку от `develop`:

```bash
# Переключитесь на develop и обновите её
git checkout develop
git pull origin develop

# Создайте новую ветку с корректным именем
git checkout -b feature/nazvanie-funkcii
```

### 2. Соглашения по именованию веток

Используйте префиксы для типов изменений:

- **`feature/`** - новая функциональность (например: `feature/dark-theme`, `feature/user-auth`)
- **`fix/`** - исправление бага (например: `fix/header-scroll`, `fix/form-validation`)
- **`refactor/`** - рефакторинг кода без изменения функциональности (например: `refactor/api-layer`)
- **`docs/`** - изменения в документации (например: `docs/readme-update`)
- **`style/`** - изменения форматирования, стилей (например: `style/button-colors`)
- **`test/`** - добавление или изменение тестов (например: `test/button-component`)
- **`chore/`** - обновление зависимостей, конфигурации (например: `chore/update-deps`)

**Формат имени**: `тип/краткое-описание-на-английском` (все маленькими буквами, слова через дефис)

**Примеры хороших имен:**

- ✅ `feature/user-authentication`
- ✅ `fix/header-navigation-issue`
- ✅ `refactor/api-error-handling`
- ✅ `docs/contributing-guide`

**Примеры плохих имен:**

- ❌ `my-new-feature` (нет префикса)
- ❌ `feature_new` (используется подчеркивание)
- ❌ `FEATURE/BIG-FEATURE` (заглавные буквы)

### 3. Разработка

Работайте в своей ветке, делайте коммиты:

```bash
# Внесите изменения
# ...

# Добавьте изменения
git add .

# Сделайте коммит с понятным сообщением
git commit -m "feat: добавлена поддержка темной темы"

# Отправьте ветку на remote
git push origin feature/nazvanie-funkcii
```

### 4. Создание Pull Request

1. Перейдите на GitHub и создайте Pull Request
2. **Base branch**: `develop` (всегда!)
3. **Compare branch**: ваша ветка с изменениями
4. Заполните шаблон PR (автоматически подставится из `.github/pull_request_template.md`)

### 5. Ревью и мердж

- После получения одобрения PR мерджится в `develop`
- **Ветка автоматически удаляется** после мерджа (можно включить в настройках GitHub)
- Никогда не мерджите PR напрямую в `main`!

### 6. Релиз в продакшн

После накопления достаточного количества изменений в `develop`:

```bash
# Переключитесь на main и обновите
git checkout main
git pull origin main

# Смерджите develop в main
git merge develop

# Отправьте изменения
git push origin main
```

Или создайте PR `develop` → `main` для более контролируемого процесса.

## 📝 Соглашения по коммитам

Используйте [Conventional Commits](https://www.conventionalcommits.org/) формат:

```
<тип>[опциональная область]: <описание>

[опциональное тело]

[опциональная нижняя часть с метаданными]
```

**Типы коммитов:**

- `feat:` - новая функциональность
- `fix:` - исправление бага
- `docs:` - изменения в документации
- `style:` - форматирование, стили (не влияет на код)
- `refactor:` - рефакторинг кода
- `test:` - добавление/изменение тестов
- `chore:` - обновление зависимостей, конфигурации
- `perf:` - улучшение производительности
- `ci:` - изменения в CI/CD

**Примеры:**

- `feat(theme): добавлена поддержка темной темы`
- `fix(header): исправлена прокрутка навигации на мобильных устройствах`
- `refactor(api): упрощена обработка ошибок в API клиенте`
- `docs(readme): обновлена инструкция по установке`

## ✅ Чеклист перед созданием PR

- [ ] Код проходит линтинг (`pnpm lint`)
- [ ] Код отформатирован (`pnpm format`)
- [ ] Нет ошибок TypeScript (`pnpm type-check`)
- [ ] Тесты проходят (`pnpm test`)
- [ ] Коммиты следуют конвенции (Conventional Commits)
- [ ] Ветка создана от `develop`
- [ ] PR создан в `develop` (не в `main`!)
- [ ] Описание PR заполнено

## 🔄 Синхронизация локальной ветки develop

Если нужно обновить локальную ветку `develop`:

```bash
git checkout develop
git pull origin develop
```

## 🚫 Что НЕ делать

- ❌ Не коммитить напрямую в `main` или `develop`
- ❌ Не создавать ветки от `main` для разработки новых фич
- ❌ Не мерджить PR в `main` напрямую (только через `develop`)
- ❌ Не использовать `merge --no-ff` без необходимости
- ❌ Не оставлять мертвые ветки после мерджа
